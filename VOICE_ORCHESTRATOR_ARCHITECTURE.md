# Voice Orchestrator Architecture (Assembly AI + Claude + Twilio + Google TTS)

## Overview
Replacement for Vapi with full control over the conversational loop. Lower latency (~200-300ms STT), natural TTS, and custom LLM logic.

```
Twilio (SIP)
    ↓ (audio stream)
Assembly AI WebSocket (STT, 200-300ms latency)
    ↓ (transcript)
Claude API (decision logic)
    ↓ (response text)
Google Cloud TTS (synthesis)
    ↓ (audio)
Twilio (send to debtor)
    ↓
MongoDB (log call, transcript, decisions)
```

## Components

### 1. **Twilio Integration** (Phone line)
- Receives/initiates calls via TwiML
- Streams audio to WebSocket
- Receives audio from orchestrator, plays to debtor

### 2. **Assembly AI WebSocket** (STT)
- Real-time speech-to-text streaming
- Latency: 200-300ms (vs Vapi ~1-2s)
- Model: `u3-rt-pro` (streaming universal)
- Supports Spanish (Colombian dialect)

### 3. **Claude API** (Logic)
- Takes transcript + call context
- Returns next action: ask question, confirm payment, escalate to human, etc.
- Uses Landa's cobranza estrategia (already generated by Queen)

### 4. **Google Cloud TTS** (Speech synthesis)
- Converts response text to natural Spanish audio
- Latency: ~500-800ms (acceptable for turn-taking)

### 5. **State Machine** (Call orchestration)
```
START
  ├─ Verify identity (intro greeting)
  ├─ Confirm debt details (monto, vencimiento)
  ├─ Offer payment options (payment_date, amount)
  ├─ Handle objections (loop back to decision)
  ├─ Escalate if max_intentos exceeded
  └─ END (log result to MongoDB)
```

## Database Schema

### `cobranza_calls` (new collection)
```python
{
    "_id": ObjectId,
    "user_id": str,
    "debtor_id": ObjectId,
    "call_id": str,  # Twilio call SID
    "estado": "en_curso" | "completada" | "fallida",
    "transcript": [
        {"speaker": "agent", "text": "...", "timestamp": datetime},
        {"speaker": "debtor", "text": "...", "timestamp": datetime},
    ],
    "decisions": [
        {
            "turn": int,
            "transcript_so_far": str,
            "decision": "ask_confirmation" | "offer_payment" | "handle_objection" | "escalate",
            "reasoning": str,
            "response_text": str,
        }
    ],
    "result": {
        "paid": bool,
        "payment_date": str | null,
        "amount_paid": float | null,
        "escalated": bool,
        "reason_ended": str,
    },
    "created_at": datetime,
    "updated_at": datetime,
}
```

## Flow

### Phase 1: Setup
1. User clicks "Llamar ahora" in dashboard
2. Backend validates (Ley 2300 compliance, config exists)
3. Initiates Twilio outbound call to debtor
4. Twilio connects to TwiML webhook (FastAPI endpoint)

### Phase 2: Call Active (WebSocket)
1. TwiML webhook upgrades to WebSocket
2. Twilio streams audio to Assembly AI
3. Assembly AI streams transcript to Claude logic
4. Claude decides response
5. Response goes to Google TTS
6. TTS audio streams back to Twilio
7. Twilio plays to debtor
8. Loop continues until end condition

### Phase 3: Call End
1. Debtor hangs up OR agent escalates
2. Log final transcript + decisions to MongoDB
3. Update debtor estado based on outcome
4. Fire completion webhook

## Latency Breakdown (per turn)

| Component | Latency |
|-----------|---------|
| Assembly AI (STT) | 200-300ms |
| Claude (API call) | 500-1000ms |
| Google TTS (synthesis) | 500-800ms |
| Network overhead | 100-200ms |
| **Total per turn** | **1.3-2.3 seconds** |

Compare to Vapi: ~2-3s per turn + feels robotic due to pre-recorded prompts.

## Fallback Strategy (Optional)

If Assembly AI service degradation:
1. Detect Assembly AI errors
2. Fall back to Vapi call for that session
3. Log fallback event for monitoring
